Diagram 02

Market Lifecycle

From AI-generated event → market creation → trading → resolution → redemption — fully autonomous.

1

🤖 AI Creates Event

Gemini generates a unique market question. CRE cron handler calls MarketFactory.createMarket() on-chain.

2

💧 Liquidity Seeded

Factory transfers collateral. Market deploys YES/NO tokens and seeds the constant-product AMM pool.

3

📈 Active Trading

Users mint complete sets, swap YES↔NO via AMM, and add/remove liquidity. Agents trade via the agents-workflow.

4

⏰ Market Closes

At close time, trading halts. Resolution time gate ensures no premature settlement.

5

⚖️ AI Resolves

CRE calls Gemini with search grounding. AI returns YES/NO/INCONCLUSIVE with source URL and confidence score.

6

✋ Dispute Window

Users can dispute. If disputed, CRE re-adjudicates with evidence. If inconclusive, manual review kicks in.

Diagram 03

Two-Workflow CRE Architecture

Operational automation and agent trading are architected as isolated CRE deployments — independent keys, configs, and failure domains.

⚙️

market-workflow

Operational Automation
  • CRON resolveEvent — AI-powered resolution
  • CRON createEventHelper — AI market creation
  • CRON marketFactoryBalanceTopUp
  • CRON syncCanonicalPrice — CCIP broadcast
  • CRON arbitrateUnsafeMarket
  • CRON adjudicateExpiredDisputes
  • HTTP sponsorUserOpPolicy
  • HTTP executeReport
  • HTTP fiatCredit
  • LOG ethCreditFromLogs
🤖

agents-workflow

Agent Trading Engine
  • HTTP agentPlanTrade — validate intent
  • HTTP agentSponsorTrade — session auth
  • HTTP agentExecuteTrade — payload + submit
  • HTTP agentGeminiAutoTrade — AI one-shot
  • HTTP agentRevoke — terminate session
  • Gemini 2.5 Flash decides action + amount.
    Bounded by policy caps.
    Routed through full security pipeline.

🔀 Why Two Workflows?

Architecture Decision
🔑

Separate Auth Keys

Market ops keys can't trigger agent trades and vice versa

🧱

Isolated Failures

Agent misconfiguration can't break market resolution

🚀

Independent Deploys

Upgrade agent logic without touching market automation

📋

Separate Policies

agentPolicy vs sponsorPolicy vs executePolicy — each scoped

Diagram 05

Six-Layer Security

Defense-in-depth: every agent trade passes through 6 independent validation layers. Even with all off-chain layers compromised, on-chain checks still block unauthorized activity.

1

🔑 HTTP Authorized Keys

CRE trigger gate — only requests signed by registered workflow keys reach the handlers

CRE trigger
2

📝 EIP-712 Session Signatures

Sponsor policy verifies typed-data grant signatures from user wallet + session key

CRE handler
3

🔒 One-Time Approval + Nonce Replay

Firestore stores approval records — each can only be consumed once, nonces prevent replay

CRE state
4

📋 Execute Policy Allowlist

Only whitelisted routerAgent action types can be submitted on-chain

CRE handler
5

⛓️ On-Chain _authorizeAgent()

Smart contract checks: permission enabled · not expired · action in mask · amount within cap

Smart contract
6

💰 Router Balance & Risk Checks

Market allowlist verified · user balance sufficient · risk exposure cap enforced

Smart contract
